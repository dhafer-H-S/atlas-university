## What This Project Is

This is a **university mid-term C++ systems project** — NOT a web app, NOT a data science project, NOT a machine learning project.

**The goal:**
> Measure empirically how **array size** and **memory access stride** affect program runtime, then explain the results using the L1 / L2 / L3 cache hierarchy and the concepts of **spatial locality** and **temporal locality**.

The project produces:
1. A **C++ benchmarking executable** (`./cache_lab`) that runs 5 memory experiments
2. **CSV output files** in `results/` with measured timing data
3. **PNG plots** generated by `scripts/plot_results.py`
4. A **written report** (`report/report.md`)
5. A **Marp presentation** (`presentation/slides.md`)

## The 5 Experiments

### Experiment 1 — Stride Access Pattern (`stride_bench.cpp`)
- **What:** Fixed 64 MB `int` array; access every `stride`-th element (stride = 1, 2, 4, 8, 16, 32, 64, 128, 256, 512, 1024 elements)
- **Measures:** ns/access vs stride in elements
- **Key result:** Latency jumps at stride = 16 elements = 64 bytes (one cache line width)
- **Output:** `results/stride.csv` → columns: `stride_elements, stride_bytes, ns_per_access`

### Experiment 2 — Working Set Size (`workingset_bench.cpp`)
- **What:** Sequential scan (stride=1) of arrays from 1 KB to 256 MB (powers of 2)
- **Measures:** ns/access vs array size
- **Key result:** Three latency plateaus matching L1 (< 32 KB), L2 (< 256 KB), L3 (< 6 MB), then RAM
- **Output:** `results/workingset.csv` → columns: `array_bytes, array_kb, ns_per_access`

### Experiment 3 — Matrix Traversal (`matrix_bench.cpp`)
- **What:** NxN `double` matrix summed in row-major order vs column-major order (N = 64 to 4096)
- **Measures:** ns/access for both patterns + slowdown ratio
- **Key result:** Col-major is 1.5–20× slower depending on matrix size vs L3
- **Output:** `results/matrix.csv` → columns: `N, matrix_bytes, rowmajor_ns, colmajor_ns, slowdown_ratio`

### Experiment 4 — Pointer Chasing (`linkedlist_bench.cpp`)
- **What:** N nodes of 64 bytes (= 1 cache line) with shuffled (random) `next` pointers; sweep list size 4 KB → 256 MB
- **Measures:** ns/hop vs list size
- **Key result:** Random access defeats the hardware prefetcher → much higher latency than sequential for large sizes
- **Node struct:** `struct alignas(64) Node { Node* next; char pad[56]; };`
- **Output:** `results/linkedlist.csv` → columns: `list_bytes, list_kb, ns_per_hop`

### Experiment 5 — Memory Bandwidth (`bandwidth_bench.cpp`)
- **What:** Array copy (`b[i] = a[i]`) and array sum (`sum += a[i]`) of `double`, sweep 4 KB → 256 MB
- **Measures:** GB/s for copy and sum vs array size
- **Key result:** L1 bandwidth >> L3 bandwidth >> DRAM bandwidth; sharp drops at cache boundaries
- **Output:** `results/bandwidth.csv` → columns: `array_bytes, array_kb, copy_gb_per_s, sum_gb_per_s`

## Key Technical Choices (Why We Did Things This Way)

| Choice | Reason |
|---|---|
| `volatile` sink accumulator | Prevents compiler dead-code elimination of benchmark loops under `-O2` |
| Best-of-5 timing | Removes OS scheduling jitter; minimum time = maximum hardware performance |
| Warm-up pass | Ensures OS pages are mapped before timing; isolates cache effects from page-fault cost |
| `-O2` (not `-O3`) | Enables realistic optimisation without auto-vectorisation/loop-unrolling that would skew results |
| `alignas(64)` on Node | Forces each linked-list node to its own cache line; prevents false spatial locality |
| `std::chrono::high_resolution_clock` | Nanosecond precision; portable across macOS/Linux |
| Fisher-Yates shuffle for linked list | Creates a true random permutation; ensures every pointer hop goes to an unpredictable address |

## Core Concepts You Need to Know to Help Us

### Cache Hierarchy
```
CPU Registers  < 1 KB   ~0.3 ns   managed by compiler
L1 Cache        32 KB   ~1-2 ns   ← hot data lives here
L2 Cache       256 KB   ~4 ns
L3 Cache         6 MB   ~12 ns
Main RAM        > 8 GB  ~60 ns    ← cold data comes from here
```

### Cache Line
- The CPU always fetches **64 bytes at once** — never just 1 element
- This is a "cache line" — the atom of memory transfer
- `int arr[16]` fits in exactly 1 cache line

### Spatial Locality
- Accessing `arr[0]` loads the whole cache line → `arr[1..15]` are free hits
- Sequential loops exploit this fully
- Col-major matrix access wastes it (loads 64 bytes, uses 8 bytes, discards the rest)

### Temporal Locality
- Loop counters (`i`, `sum`) are reused every iteration → stay in L1/registers
- Hot inner-loop data should fit in L1 (< 32 KB)

### Hardware Prefetcher
- Detects regular stride patterns and loads the next cache line early
- **Cannot predict pointer-chasing** — next address is unknown until current load completes

### Load-Use Dependency Chain (pointer chasing)
```cpp
p = p->next;   // must WAIT for this load to finish
               // before knowing which address to fetch next
               // → all latency is serialised, no overlap possible
```
